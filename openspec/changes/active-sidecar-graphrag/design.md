# Design: Active Sidecar Architecture

## Architecture Overview

```
[ User / MCP ] 
      |
      v (Ingest / Search)
+-----------------------+
|     Core Engine       | ---- [ Event Channel ] ----+
| (EdgeQuake Funnel)    |                            |
+-----------------------+                            |
      |                                              |
      v (Sync Write)                                 v (Async Listen)
+-----------------------+                  +-----------------------+
|    SQLite (Master)    | <--------------- |    Graph Sidecar      |
| - Docs / Entities     |   (Sync Enrich)  | - petgraph In-Memory  |
| - COMMUNITIES (New)   |                  | - Clustering logic    |
+-----------------------+                  +-----------------------+
                                                     |
                                                     v
                                           +-----------------------+
                                           |  Summarizer Sidecar   |
                                           | - Prompts NuExtract   |
                                           | - Gen Community Reps  |
                                           +-----------------------+
```

## Core Components

### 1. Ingestion Event Channel
The `IngestionPipeline` will emit an event whenever a new entity or relationship is added.
- **Type**: `tokio::sync::broadcast::Sender<KnowledgeEvent>`
- **Event**: `KnowledgeEvent { entity_id, name, namespace }`

### 2. Graph Sidecar (`GraphObserver`)
A long-running thread that maintains a `petgraph::Graph` of the entire system.
- **Incremental Clustering**: Uses a Label Propagation algorithm to assign nodes to `community_ids`.
- **Level Management**: Tracks hierarchy (small clusters vs. large themes).

### 3. Community Sidecar (`CommunityService`)
Reacts when a community's membership changes significantly.
- **Summarization**: Prompts the LLM to "Summarize the key facts about this group of entities."
- **Persistence**: Writes the summary back to the `communities` table in SQLite.

## Schema Updates
New `communities` table:
- `id`: Unique cluster ID.
- `title`: Generated by LLM (e.g., "Boston Winter Storms").
- `summary`: High-level thematic text.
- `level`: Hierarchical depth (0=local, 1=global).
- `metadata`: JSON blob of member counts and timestamps.
